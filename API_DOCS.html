Part 1: Feature Implementation Checklist

Detailed justification mapping the assignment's core requirements directly to the codebase implementation.

1. User and Role Management

Implemented via Prisma Relational Schema & Users API.
A deeply structured relational database model was built using Prisma with a dedicated Role enum (ADMIN, ANALYST, VIEWER). The management API (src/routes/users.routes.ts) empowers Admins to view all users, update their roles (PATCH /users/:id/role), and suspend users (PATCH /users/:id/status by flipping the isActive boolean). Suspending a user immediately blocks them at the authentication layer.

2. Financial Records CRUD

Implemented via records.routes.ts.
A comprehensive REST API was built over a Record table that maps one-to-many with Users. All standard operations are supported securely: Create (maps incoming JSON to prisma.create), Read (findMany), Update (update), and Delete (delete). Strict ownership and relational data integrity guarantees no orphaned records.

3. Record Filtering (by date, category, type)

Implemented dynamically in the GET /records endpoint.
Instead of pulling all data into memory to filter, the backend dynamically constructs a Prisma where constraints object based on intercepted Query Parameters (e.g., ?type=INCOME&startDate=2024-01-01). Prisma translates this safely into optimal, raw SQL WHERE clauses, ensuring high query speeds and resilience against SQL injection.

4. Dashboard Summary APIs

Implemented natively using Database-Engine Aggregation.
Instead of fetching thousands of raw rows and calculating totals via Javascript loops, the GET /dashboard/summary endpoint utilizes Prisma's native .aggregate() and .groupBy() computing functions. This offloads the heavy mathematical lifting (summing incomes, expenses, and category totals) directly to the PostgreSQL engine, resulting in instantaneous responses regardless of ledger scale.

5. Role Based Access Control (RBAC)

Implemented via Express "Chain of Responsibility" Middleware.
Security is handled defensively at the edge. First, auth.middleware.ts decodes the stateless JWT to verify identity and account active-status. Second, role.middleware.ts intercepts the request to verify the user matches the array of allowed roles on that specific route (e.g., requireRole(['ADMIN'])). Unauthorized attempts are instantly rejected with a 403 Forbidden.

6. Input Validation and Error Handling

Implemented using Zod schemas and Global Error Handlers.
To prevent invalid database states, every incoming POST/PUT/PATCH request is filtered through strict Zod schemas (src/schemas/record.schema.ts). A generic validation middleware intercepts and throws a 400 Bad Request automatically if the payload doesn't match the required type or shape. Unhandled promise rejections or server crashes are caught by a global error handler ensuring standardized JSON output instead of crashing the server.

7. Data Persistence

Implemented using deployed Serverless PostgreSQL on Supabase & bulk seeding.
The backend utilizes a highly resilient cloud PostgreSQL database hosted on Supabase using an IPv4 Session Pooler connection — ensuring zero cold-start issues and persistent always-on availability. To ensure the API is fully testable immediately, a massive algorithmic seeding script (prisma/seed.ts) was engineered. It generates and writes 1,000 randomized, realistic financial records distributed across the mock users via robust createMany bulk insertions.

Part 2: Technical Architecture & Decisions

This section outlines the engineering decisions, architectural patterns, and technical stack choices made during the development of this backend.

1. System Architecture

  • Routing Layer: Express.js handles HTTP routing. Routes are decoupled into modules (Auth, User, Record, Dashboard) to keep the entry point clean.
  • Middleware Layer: A defensive "chain of responsibility" pattern using auth.middleware (JWT verification), role.middleware (Authorization), and validate.middleware (Zod request parsing).
  • Persistence Layer: Prisma ORM provides a type-safe interface mapped directly to a cloud PostgreSQL database.
  • Global Error Handler: Uses express-async-errors to catch unhandled promise rejections and normalize them into a standard, secure JSON error response format without leaking stack traces.

2. Security & Identity Management

  • Password Hashing: Utilizes bcrypt with a salt round of 10 to protect against rainbow table attacks.
  • Token-Based Auth: Stateless authentication using JSON Web Tokens (JWT).
  • RBAC (Role-Based Access Control): Permissions are strictly enforced horizontally across endpoints via role arrays (e.g., ['ADMIN', 'ANALYST']). Modifying a user account to 'inactive' blocks them at the middleware layer on their very next request.

3. Data Modeling & Validation

  • Primary Keys: Uses UUID (v4) for all primary keys instead of auto-incrementing integers to prevent ID guessing (Insecure Direct Object Reference attacks).
  • Relational Integrity: Built a strict One-to-Many relationship where User owns Record.
  • Zod Enforcement: Input validation intercepts the request natively. Zod schemas are used to infer TypeScript types, ensuring the controller logic never processes undefined or risky properties.
  • Dashboard Aggregations: The Dashboard leverages database-engine calculations (Prisma's aggregate and groupBy algorithms) rather than mapping arrays in-memory, ensuring performance remains lightning fast even at immense scale.

GET /dashboard/summary ADMIN, ANALYST, VIEWER

Returns aggregated, high-level financial statistics computed by the database engine.

Response Returns:
  • totalIncome: Sum of all income records.
  • totalExpense: Sum of all expense records.
  • netBalance: Total Income minus Total Expense.
  • categoryTotals: Breakdown of spending/earning mapped per category.
  • recentActivity: The 5 most recently created transactions.

GET /records ADMIN, ANALYST

Fetches all financial transactions. Supports advanced filtering via exact matches and date ranges.

Optional Query Parameters:
  • type: INCOME or EXPENSE
  • category: String (e.g., 'Salary', 'Rent'). Note: This search is robustly case-insensitive.
  • startDate: ISO Date string (e.g., 2024-01-01)
  • endDate: ISO Date string (e.g., 2024-12-31)
  • paymentMethod: CARD, CASH, TRANSFER, CRYPTO
  • status: COMPLETED, PENDING, FAILED
  • currency: 3 letter string (e.g., USD)

POST /records ADMIN ONLY

Creates a new financial entry in the ledger.

Request Body (Required Fields): 
{
  "amount": 1500.50,
  "type": "INCOME",
  "category": "Project Payment",
  "date": "2024-03-25T10:00:00Z",
  "paymentMethod": "TRANSFER",
  "currency": "INR",
  "notes": "Payment for Q1 Consulting"
}

PUT /records/:id ADMIN ONLY

Updates an existing record. All fields in the creation schema are accepted but optional.

DELETE /records/:id ADMIN ONLY

Permanently deletes a financial record from the database.

GET /users ADMIN ONLY

Returns a list of all registered users with their current status and roles.

PATCH /users/:id/role ADMIN ONLY

Upgrades or downgrades a user's permissions level.

Request Body: 
{ 
  "role": "ANALYST" 
}

PATCH /users/:id/status ADMIN ONLY

Activates or deactivates a user account. Deactivated users are blocked from consuming endpoints.

Request Body: 
{ 
  "isActive": false 
}
